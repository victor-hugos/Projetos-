# Decompondo a Estrutura

## Responsável pela Decomposição

### Líder Técnico/Time Técnico

## Premissa

Clareza sobre o **Planejamento de Execução** (Plano de Trabalho). É desejável que o plano de trabalho tenha sido validado previamente pelo cliente para que os objetivos do **Plano de Ação** estejam alinhados com o valor a ser entregue. 

## Decompondo uma estrutura


:::info
Nos passos a seguir, estão ilustrados exemplos de um projeto já existente (Ministério da Saúde), caso o seu projeto seja diferente, por favor entre em contato com seu Administrador do Jira para criação do Projeto e Roteiro Avançado.

:::

Dando continuidade ao exemplo do tópico anterior:

 ![Figura 1: Itens de trabalho (macro)](/api/attachments.redirect?id=5c777e4c-5008-4e5e-b894-1cc3b5a5cdae)

No exemplo acima, foram criadas macroatividades, que segundo a visão negocial do Gerente de Projetos, entregariam o valor contratado na Sprint.

Para continuar com a criação da estrutura no detalhe, é necessário agora a participação de outros atores. A continuidade agora é com o Líder Técnico, que após empoderar-se da demanda, deverá avaliar a estrutura básica, podendo incluir outros itens que julgar necessário para o alcance dos objetivos.

Considere no exemplo abaixo que o Líder Técnico enxergou a necessidade de duas macroatividades além das que já haviam sido criadas pelo Gerente de Projetos:

 ![Figura 2: itens de trabalho (macro)](/api/attachments.redirect?id=c14705c6-e05f-4669-8f75-006e9780c16f)

Passamos então para a decomposição das tarefas, agora com a participação do Time Técnico. Nesse passo a equipe deverá criar os itens de trabalho em nível de subtarefa, na menor unidade de trabalho possível. Isso dará a visão real de esforço e facilitará na gestão do progresso das atividades.

Por exemplo:

 ![Figura 3: itens de trabalho (micro)](/api/attachments.redirect?id=42e890e0-6d2f-4bd2-8d70-1e58dadeb64c)

 ![Figura 4: itens de trabalho (micro)](/api/attachments.redirect?id=6342bae7-d4d4-4208-a1af-5aa8e1cc3a17)

No exemplo acima, para a macroatividade “Backlog do Produto“, foram criadas as subtarefas: Elaborar backlog do produto, Validar backlog do produto, Priorizar backlog da sprint, isso pelo entendimento que estas atividades são necessárias para entrega de uma parte do trabalho previsto na Sprint que é a entrega do Backlog do Produto (o mesmo formato deve ser adotado inclusive para as entregas “formais“ do contrato, no caso em questão o backlog é um entregável previsto para o faturamento da demanda).

Também, na Funcionalidade 1, foram criadas as subtarefas: Mapear jornada do usuário, Elaborar protótipo, Especificar história do usuário, Desenvolver front end, Desenvolver back end, Integração front end x back end, também pela conclusão de que são atividades necessárias para entrega da funcionalidade.

Isso deve ser feito para toda macroatividade prevista no planejamento, criando assim uma estrutura geral para o atendimento do serviço.

Uma vez decompostas todas as microatividades, todo o time deve ainda participar do planejamento, programando data de início e fim para cada item.

Uma maneira fácil e visual de realizar a programação é através do próprio Gráfico de Gantt. Contraindo-se os campos, e apenas com um clique sobre a linha do tempo, é possível programar datas para os itens individualmente.

 ![Figura 5: programando datas pelo diagrama de Gantt](/api/attachments.redirect?id=bc99b229-d17b-441a-bb71-b0bf05a06b47)

A programação das datas pode ser feita desta forma, ou simplesmente preenchendo os valores diretamente nos campos.

 ![Figura 6: programando datas pelo preenchimento dos campos](/api/attachments.redirect?id=da29e46b-d348-4517-b1e0-af2e6d465643)

Para conclusão desta etapa, é necessário que todos os itens de trabalho estejam vinculados a um projeto, com data de início e de término, bem como atribuídos para algum responsável, conforme descrito no tópico “Criação da Estrutura“.
